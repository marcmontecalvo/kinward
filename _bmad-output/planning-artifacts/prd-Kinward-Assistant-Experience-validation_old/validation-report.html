Adversarial review

review-adversarial-general.md

Materially improved but not yet ready for architecture or epic decomposition. All seven prior high-severity findings were structurally addressed, and five prior focus areas are fully resolved. However, the group-mode fix introduces a selected-share exception that its own audience caveat makes unsafe, the privacy-classification taxonomy is now internally inconsistent across Sections 7, 8, and 10, and the new decision registers assign due dates that postdate Milestone A work that depends on them. One more focused revision pass should close the document. Counts: 0 critical, 2 high, 10 medium, 4 low.

High

Group-state `selected-share` exception is unsatisfiable under its own audience caveat

L392, L328, L248
Line 392 permits rendering "a `selected-share` that is valid for every detected audience member" while the same sentence states "presence detection is never treated as proof that every audience member is known." If detection can never establish the audience, no selected-share can ever be proven valid for everyone present — the permission is either dead text or will be implemented against detected presence and disclose selected-share content (which originates from private data) to undetected people, recreating exactly the group-mode leak the revision was written to eliminate.
Fix: In `group` state permit `household-shared` rendering only; route `selected-share` and all private content through private-device handoff, or require every detected person to hold a currently `verified` session plus explicit surface-policy opt-in, with any unverified presence downgrading to household-shared. Also state that an explicit sharing act reclassifies the shared statement per Section 8.2.
High

Privacy-class taxonomy internally inconsistent across Sections 7, 8, and 10

L291, L306, L324–331, L392, L940
Section 8.1 enumerates exactly six mandatory data classes, but Section 7.4 uses `private-to-child` (naming drift from `private-child`) and `shared-with-guardians` (absent from the enumeration; PD-02's interim is keyed to it), and Section 7.3's `teen` class has no stated mapping — so the group-state prohibition, which names only `private-person` and `private-child`, does not literally cover teen private data. FR-029/FR-032/NFR-003/NFR-004 tests cannot be written consistently against three divergent taxonomies.
Fix: One normative mapping: `private-to-child` = `private-child`; `shared-with-guardians` = a constrained `selected-share` naming the guardians; state which class teen private content uses; rephrase the group prohibition to close over the complete class set (e.g., "everything except `household-shared`"); align PD-02's wording.
Medium

UJ-3 contradicts the coordination-statement alternative in Section 4.1.2

L102–115 vs L194
Section 4.1.2 allows the shared fixture to be a private topic represented by an approved household-safe coordination statement; UJ-3 still permits the generic shared item "only when the topic is household-shared," forbidding that branch, and "shared/unshared" is undefined for it.
Fix: Define "shared" for the slice as household-shared or having an approved Section 8.2 coordination statement; amend UJ-3's shared-display outcome to permit the generic item when either condition holds.
Medium

Decision-register due dates postdate Milestone A work that depends on them

L711, L494, L772, L917–922
Milestone A includes the bootstrap API (FR-002 creates account bindings), neutral memory/knowledge contracts, and provider adapters — yet AD-01 (authentication/binding), AD-02 (model-provider contract), and AD-06 (storage split) are due only at Milestone B start. Milestone A either implements ahead of its governing decisions or the milestone assignments are wrong; the salvage rule requires contracts before migration.
Fix: Pull the contract-shaping parts of AD-01/AD-02/AD-06 forward to Milestone A start (or split each into an A-scoped contract decision and a B-scoped mechanism decision), or move the dependent Milestone A content to B.
Medium

Member (non-administrator) post-restore access is neither specified nor verified

§14.3 L629–630 · FR-085/086
The new contract gives the administrator a documented recovery procedure and verifies it, but members get one sentence with no required procedure; all pre-restore invitations are invalid and no supersession rule covers a profile's orphaned prior account record. FR-086 never verifies member re-access, so INV-8 can pass with every non-admin locked out.
Fix: Require a documented member re-access procedure (post-restore invitation to the existing profile or portable member recovery material), define supersession of prior credentials, and extend FR-086 to verify member re-access for at least one non-administrator profile.
Medium

Autonomous-authority machinery: Milestone C or D scope is ambiguous

L408–419, L552, L559, L264–272
FR-052 (Milestone C) requires "every authority rule in Section 11," including the full `autonomous` grant contract, while FR-045 and Section 6.4 mandate autonomous delivery stays disabled until D. The PRD never says whether C builds the (disabled) grant mechanism or FR-052 is satisfied vacuously.
Fix: State that during Milestone C no `autonomous` grant can be created or activated (Section 11 autonomous rules enforce "always deny") and the grant mechanism, controls, and Section 6.5 measurement ship in Milestone D.
Medium

Guardian review of child data contradicts PD-02; child account path undefined

§7.4 L306–307 · FR-032 · PD-02 · UJ-7
Section 7.4 still grants administrators review "necessary for care, safety, school…" without precedence over `private-to-child`, a necessity principal, or audit trail — contradicting PD-02's interim; FR-032 (Milestone C) requires implementing Section 7 as written, which is unimplementable while PD-02 is open. UJ-7 depends on a child account no FR creates.
Fix: Subordinate 7.4's review sentence to PD-02's resolution; add precedence, decision principal, minimum-necessary disclosure, and append-protected audit for exceptional access; add or explicitly defer a child-account onboarding/binding FR.
Medium

Action authority still has no grant/approval/revocation principal matrix

§11 L410–422 · FR-053
Carried from the prior review: the PRD never states who may grant or revoke a capability maximum or bounded autonomy, or who is an eligible approver when an action affects another person; no AD/PD entry owns the gap.
Fix: Add a grant/approval/revocation matrix keyed by privacy class, represented person, action category, affected person, and integration ownership with execution-time validation — or minimally a register entry with a safe interim.
Medium

Inference-provider degradation still less defined than memory and Home Assistant

L135, L159, L222–226, L918
AD-02's interim helps, but no FR defines behavior when no model provider is configured, whether accepted requests fail or hang on outage, or the minimum inference capability required for the Milestone C "working assistant conversation path" gate.
Fix: Add an FR mirroring FR-027/FR-067 for the inference capability and define the minimum configured inference capability for pilot entry; extend UJ-8.
Medium

Specialist assistants still have no lifecycle requirement or milestone

§3.2, §9.1, FR-010, NFR-004
Carried, unchanged: no FR lets an owner create, select, retire, or delete a specialist and no milestone or explicit deferral covers the lifecycle, so NFR-004 tests a boundary no requirement brings into existence.
Fix: Assign a minimum lifecycle to a milestone or defer it explicitly in Section 4.3, preserving multi-assistant schema compatibility and forbidding implicit private-memory merges.
Medium

Prepared approvals and activity records omit "what information was used"

L416, L560, L547
Carried, unchanged: FR-053's record lists actors, authority, approval, integration, results, and undo — but not the privacy-filtered source categories or correctable facts the action relied on.
Fix: Require prepared approvals and final activity records to include privacy-filtered source categories and referenced user-correctable facts, excluding others' private data, secrets, prompts, and hidden reasoning.
Medium

"Exactly one household" still lacks enforcement-path requirements

FR-001, FR-076, INV-1
INV-1 and FR-076 improve this, but no requirement rejects a second household through APIs, jobs, imports, or restore-to-populated paths, and nothing prohibits tenant-routing or commercial-role fields in Kinward contracts.
Fix: Extend FR-001 to require rejection on every path and prohibit tenant/entitlement/support-operator fields absent an approved justification.
