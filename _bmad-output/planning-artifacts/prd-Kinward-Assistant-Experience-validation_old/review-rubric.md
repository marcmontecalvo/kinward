# PRD Quality Review — Kinward Assistant Experience (re-validation)

Reviewed: `_bmad-output/planning-artifacts/prd-Kinward-Assistant-Experience.md` (2026-07-13 revision, 965 lines) against the four authoritative sources (product brief, UX design specification, single-household pivot and rebuild plan, salvage matrix) and the prior rubric review. `docs/pivot/migration-status.md` was not treated as product authority and is no longer referenced by the PRD.

## Overall verdict

**Strong and close to finalizable.** The revision genuinely closes every high-severity defect from the prior review: the non-authoritative source is gone, Milestone A is evidence-gated against the salvage-matrix acceptance rule, the five-context frontend-foundation gate restores the authoritative cross-surface validation shape, both decision registers now carry owners/status/due/affected-IDs/safe-interim behavior, and Section 17.4 provides complete per-requirement traceability that survived mechanical verification with zero errors. What remains is a set of medium-severity residuals — an undefined pilot metric dictionary, operation-level Milestone C scope for Kinward Control, a missing teen journey, one qualitative NFR, and a handful of qualifier escape hatches — none of which blocks architecture or epic work from starting, but which should close before the PRD is marked final per its own Section 20 criteria.

## Decision-readiness — strong

Sections 18 and 19 (lines 911–947) are now real decision registers: 15 architecture decisions (AD-01–AD-15) and 7 product decisions (PD-01–PD-07), each with owner, status, due milestone, affected requirement IDs, and a safe interim behavior that is genuinely conservative ("No implementation of invitation, session, or recovery flows until resolved" for AD-01; "Performance gates cannot be marked passed until the reference environment is defined" for AD-15; "Exclude credential material; every integration requires reauthorization" for PD-06). The closing rule (line 947) forbids silently deciding these inside stories, and Section 20 (line 959) requires resolution or explicit accepted-open status before finalization. All 22 decisions are honestly marked Open rather than smoothed over — a decision-maker can see exactly what blocks Milestone B (eight decisions due at B start) and what conservative behavior applies meanwhile. Trade-offs elsewhere are stated as decisions (Section 4.4 exclusions, Section 3.6 deferred live tablet/voice with an explicit mock-gate compensation).

No findings.

## Substance over theater — adequate

The document overwhelmingly states enforceable behavior: deterministic shared-surface states with per-state outcomes (Section 10), a five-level action-authority policy with unknown-result and stale-state rules (Section 11), an observable request lifecycle (12.1), a concrete backup/restore contract including post-restore account access (Section 14.3). The Section 10 `group` state is now categorical (line 392): "categorically prohibits disclosure of `private-person` and `private-child` data and of anything derived from private memory," bans render-time private-memory queries, and states "presence detection is never treated as proof that every audience member is known" — this closes the prior gap cleanly. NFR-012 now mandates the fallback ("idempotent where supported **or protected against duplicate execution**", line 666).

### Findings
- **medium** Residual qualifier escape hatches (§8.2 line 337, §12.1 line 440, FR-048 line 555, NFR-016 line 670) — Four qualifiers still permit materially different implementations: derived-data invalidation applies "where the product can identify the dependency" without mandating dependency registration (though AD-07 now owns the design and its interim behavior covers the external-provider branch); cancellation "must stop further model output and prevent any not-yet-submitted mutation" but never defines the terminal state for an already-submitted mutation; FR-048 requires detecting "relevant exceptions" with no relevance predicate (PD-04 pins the pilot to calendar-change briefing only, which contains but does not define it); NFR-016 says "retry-bounded" with no product-visible bound or terminal status (AD-11 owns the design). This is meaningfully reduced from the prior review — the remaining risk is that AD-07/AD-11 resolutions could be judged compliant against text that never states the mandatory floor. *Fix:* one sentence each: require a dependency record at derivation time; define the cancellation terminal state per submission stage (e.g. submitted mutations proceed to `completed`/`unknown` and are reported, never silently orphaned); state the relevance inputs and correction loop for FR-048; require a declared retry limit and terminal user-visible job status.

## Done-ness clarity — adequate

Release boundaries, deterministic policies, and most NFRs create real pass/fail conditions (NFR-017–NFR-022 carry p95/ms thresholds; Section 4.1.2 enumerates eight testable slice conditions including the negative case that "the unshared private topic is entirely absent from shared-display responses and payloads"). The perf reference environment, previously unmoored, is now managed by AD-15 with a blocking interim behavior. Remaining gaps:

### Findings
- **medium** Pilot metrics still lack reproducible measurement definitions (§6.1–6.4, lines 240–264) — "sampled assistant sessions," "sampled cross-surface topic continuations," "marked useful, acted upon, or left undisputed," and "sampled meaningful external actions" have targets but no event schema, denominator, sampling method, minimum sample size, or evaluator. Milestone C's content includes "metric instrumentation" (line 723) but nothing defines what is instrumented. The same pilot data can still yield different pass/fail results. *Fix:* add a metric dictionary (event definition, denominator, minimum N, sampling, exclusions, who judges "useful"/"undisputed") for every percentage measure in 6.1–6.5.
- **medium** NFR-024 has no objective pass condition (line 686) — it defers to "UX-specified room-distance readability and touch-target requirements," but the authoritative UX source says only "Large touch targets" and "Readable at room distance" (UX §Interaction) with no dimensions, distances, or test protocol. The a11y-audit verification method in the table (line 880) cannot be executed against this. *Fix:* state minimum touch-target size, reference viewing distances per room profile, and type-size/contrast criteria, or assign the gap a PD/AD entry due before Milestone C.
- **low** Milestone C exit gate requires Section 6 conditions to be "measurable," while Section 6 requires them to be "met" during a 30-day pilot (lines 236 vs 725) — the PRD never says what happens if the pilot misses a threshold after the exit gate passes (does Milestone C reopen? does Milestone D block?). *Fix:* one sentence stating the pilot's gating relationship — e.g., Milestone D entry requires the Milestone C pilot conditions met, not merely measurable.

### Findings

### Findings
- **medium** Teen policy still has no end-to-end journey — Section 7.3 (lines 291–299) defines a distinct `teen` privacy class with guardian-visible categories and default approval rules, and FR-032/NFR-004 require tested boundaries, but the nine journeys cover adults (UJ-1–UJ-6, UJ-9), a child (UJ-7), and degraded providers (UJ-8); UJ-7's protagonist is a child and the 17.5 coverage line for UJ-7 leans on generic supporting FRs. Epics can pass the automated-boundary bar while missing the lived teen experience (private assistance, visible guardian-category disclosure, a consequential action routed to approval, non-infantilizing tone per UX §Content Design). *Fix:* add a fictional teen journey (UJ-10 under the 17.1 governance rule) with success and refusal/handoff outcomes, mapped to FR-032, NFR-004, and the relevant action FRs.
- **low** Actor vocabulary still drifts — requirements alternate among "person" (FR-008, FR-055), "user" (FR-015), "users" (FR-016, FR-024, FR-054), "account-bearing person" (FR-009), and "member" without a normative relationship model tying profile, account, authenticated principal, role, and participant/audience together (Sections 3 and 7 define the pieces but not the model). Schema and API authors can still disagree on whether "user" means an account or any profile. *Fix:* a short normative actor-relationship paragraph in Section 3 plus a declaration of UI-only synonyms.
